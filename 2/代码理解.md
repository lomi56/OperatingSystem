#### **#include<unistd.h>// Linux 操作系统的标准接口库。**

**1、进程控制**

fork()：创建一个新进程（生孩子）。

exec() 系列：执行一个新的程序（比如在你的程序里运行 ls 命令）。

getpid() / getppid()：获取当前进程 ID 或父进程 ID。

\_exit()：立即终止进程。

**2. 文件与 I/O 操作（底层读写）**

不同于 <stdio.h> 中的 printf（带缓冲区的），<unistd.h> 提供的是直接对接内核的底层读写。

read()：从文件描述符读取数据。

write()：向文件描述符写入数据。

close()：关闭文件或管道。

lseek()：移动文件读写指针。



fork()创建子进程    int pid1=fork();     //process ID





父进程与子进程关系：

\#include<stdio.h>

\#include<unistd.h>

\#include<stdlib.h>

int main(){

&#x20;   int pid;

&#x20;   // printf("父进程\\n");

&#x20;   pid=fork();                         //当代码执行到 pid = fork(); 这一行时，你的程序从一个变成了两个。

&#x20;   if(pid==0){

&#x20;       printf("子进程\\n");

&#x20;   }

&#x20;   else{

&#x20;       printf("父进程\\n");

&#x20;   }

&#x20;   return 0;



}









执行 fork() 之前

只有一个进程在运行，我们叫它“父进程”。

执行 pid = fork(); 时

操作系统会立即创建一个新的、和父进程几乎一模一样的进程，我们叫它“子进程”。

从这一刻起，就有两个独立的进程在同时运行你的程序了。

它们都从 fork() 这一行之后继续往下执行。

父进程视角：

它拿到的 pid 是子进程的 PID（例如 12345），是一个大于 0 的数。

因此，if(pid==0) 条件不成立，它会执行 else 分支。

所以它打印出：父进程

子进程视角：

它拿到的 pid 是 0。

因此，if(pid==0) 条件成立，它会执行 if 分支。

所以它打印出：子进程





#### **#include<signal.h>**

信号处理头文件

它允许你的程序处理各种“突发事件”或“异常”，比如用户按下了 Ctrl+C、程序进行了除零运算，或者收到了操作系统的终止指令。

signal.h 定义了许多以 SIG 开头的常量，代表不同的信号：



信号常量	含义	常见触发原因

SIGINT	中断信号	用户按下 Ctrl+C

SIGSEGV	段错误	非法内存访问（如访问空指针）

SIGFPE	算术异常	除数为 0、浮点数溢出

SIGABRT	异常终止	调用了 abort() 函数

SIGTERM	终止请求	外部请求程序停止

**注：这些值本身就是int类型**



这个头文件主要提供了两个函数来操作信号：

signal(int sig, void (\*handler)(int))

用于注册信号处理函数。

告诉系统：“如果发生了 sig 信号，请执行 handler 函数。”

raise(int sig)

用于发送信号。

让程序自己产生一个信号（例如 raise(SIGABRT) 会让程序异常终止）。







#### 代码解释：



1、signal(SIGQUIT, SIG\_IGN);

signal(信号编号, 处理方式)：这是一个用于设置信号处理方式的系统调用（或库函数）。它告诉操作系统，当进程收到某个特定信号时，应该怎么做。

SIGQUIT：这是“退出”信号（Quit signal）。在终端中，通常通过按下键盘快捷键 Ctrl + \\（Control 加反斜杠）来产生这个信号。它的默认行为是终止进程，并且通常会生成一个核心转储（Core Dump）文件供调试使用。

SIG\_IGN：这是 Signal Ignore 的缩写，表示“忽略”。它是一个特殊的宏定义，作为函数的第二个参数传入，表示如果收到前面的信号，什么都不做，直接忽略。



2、

ctrl+C和ctrl+\\:



Ctrl + C 对应的是 SIGINT (Interrupt，中断信号)



作用： 这是最常规的要求程序终止的信号。

默认行为： 退出程序。

如果要忽略它： 代码应写为 signal(SIGINT, SIG\_IGN);

Ctrl + \\ 对应的是 SIGQUIT (Quit，退出信号)



作用： 这是一种“更重”的退出请求，通常用于开发者调试。

默认行为： 不仅会退出程序，还会产生一个 Core Dump (核心转储) 文件（把程序退出那一刻的内存状态保存成文件，供程序员查错用）。

如果要忽略它： 代码就是 signal(SIGQUIT, SIG\_IGN);

3、

kill(pid1, 16);     //作用是发送信号

向 ID 为 pid1 的进程发送编号为 16 的信号。

通常意味着父进程正在向子进程发送指令。

**kill和signal函数是一对，一个发送信号，一个接收信号**

4、为什么等待在信号处理的下方？

eg:

/\* 子进程 P1 代码 \*/

&#x20;       /\* 忽略 SIGQUIT 信号，只响应 16 号信号 \*/

&#x20;       signal(SIGQUIT, SIG\_IGN);//无视终端的终止操作

&#x20;       signal(16, stop\_child1);

&#x20;       printf("task1 start\\n");

&#x20;       while (1) {

&#x20;           pause();

&#x20;       }

信号接收类似于写一个库函数，在等待的过程中接收到信号就会调用，而不是返回到signal那一步。





5、两个子进程中signal(SIGQUIT, SIG\_IGN);//无视终端的终止操作   的作用

这里面的核心机制是：**当您在终端按下快捷键发送信号时，**这个信号会发送给当前前台的“整个进程组”。也就是说，**父进程和它的两个子进程会同时收到这个 SIGQUIT 信号。**



我们可以对比一下加与不加的区别：



如果不加这两行 signal(SIGQUIT, SIG\_IGN);：

当您按下 Ctrl + \\ 时：



父进程收到信号，执行 stop\_parent，把 wait\_flag 变成 0，准备继续往下走。

**两个子进程也同时收到了这个信号。**因为它们此时没有去捕获或忽略 SIGQUIT，就会执行操作系统的默认行为——**直接被强制杀掉。**

等到父进程往下运行，执行 kill(pid1, 16) 去通知子进程时，子进程其实在刚才那一瞬间早就已经死了，根本没机会执行 stop\_child1 和 stop\_child2，也就打印不出那句 "child process 1 is killed by parent!"。

加了这两行之后：

当按下 Ctrl + \\ 时：



**两个子进程收到信号但直接忽略，它们继续在 pause() 里安安静静地等。**

**只有父进程响应了 SIGQUIT 并跳出死循环。**

**父进程接下来亲手发送对应的 16 和 17 信号给依然活着的子进程。**

**子进程接收到 16 或 17 信号，执行它们各自的 stop\_child 函数，打印提示语后正常退出。**



6、wait()函数：

**它会阻塞（暂停）调用它的进程（这里是父进程），直到它的任意一个子进程结束为止。**

当一个子进程结束后，wait() 会去收集这个子进程的退出状态，并在操作系统的大表中把这个子进程彻底清理掉。

如果子进程结束了但父进程不去 wait() 它，这个子进程就会变成\*\*“僵尸进程”\*\*（Zombie Process）——它已经死了，不再运行，但仍然占用着系统里的一点进程表资源。

参数设定为 NULL 的意思：



**wait() 本来可以接收一个指针，用来保存子进程具体的死亡原因和退出码（比如是正常退出还是由于某种报错退出的）。**

传 NULL 意味着父进程：“我不在乎子进程是怎么死的，我只负责等它死掉并把它清扫干净就行了。”

为什么这里要写两遍？



因为在这个程序中，父进程通过两次 fork() 一共创建了两个子进程（pid1 和 pid2）。

每一个 wait() 只能等待并回收一个子进程。

所以父进程需要连续调用两次 wait(NULL);。第一次调用拿走先结束的那个子进程，第二次调用拿走另一个。等到这两句都执行完，说明两个子进程都干净利落地结束了。
